# Bugfix 2026-07-18: 180° heading flip at init for near-vertical CAD layouts

## Symptom

`init!` on the ram_air_kite model (OBJ-mesh `RIGID_DYNAMICS` wing, HybridWings.jl)
fails with

```
AssertionError: VSM solve failed (non-converged or non-finite) on wing 1
```

whenever the tethers' `init_stretched_len` is set to ≥ ~52 m (the export's
native length is 50 m). 51 m works; 52 m crashes — a razor-sharp boundary.
At ≥ 52 m only initial elevations ≤ ~45° converge; the ceiling stays flat at
~45° out to 150 m. This broke `test_figure_eight.jl` (150 m tether) and any
script stretching the tethers beyond ~51 m.

Ruled out during diagnosis: winch brake state, depower seed (−0.3 … 1.2 m,
including a fine 0–0.02 sweep), `aero_z_offset` (0–1 m), and VSM solver
settings (5000 iterations, relaxation 0.005, rtol 1e-3 — still failed, i.e.
genuinely out-of-range AoA, not slow convergence).

## Root cause

The wing was initialized **facing backwards**: heading = −180° instead of the
transform's requested 0°, giving AoA = 180° − 13.5° = 166.5° (tail into the
wind). No airfoil polar covers that, so the VSM solve returns non-finite
values. Measured at the failure boundary:

| | 51 m (ok) | 52 m (crash) |
|---|---|---|
| `Q_b_to_w` | `[0.993, 0, 0.118, 0]` | `[0, 0.118, 0, 0.993]` (= same ∘ 180° yaw) |
| heading | 0.0° | −180.0° |
| initial AoA | 13.5° | 166.5° |
| `va_b` | `[+12.0, 0, 2.9]` | `[−12.0, 0, 2.9]` |

Two stacked defects produced this:

### Defect 1 — degenerate tangent frame in `calc_R_t_to_w`
(`src/generate_system/helpers.jl`)

The tangent-sphere frame's azimuthal axis is built from
`normalize([-pos_y, pos_x, 0])`. In the ram export's raw CAD+stretch state
the wing sits almost exactly above the base point — the horizontal component
of its relative position is on the order of **millimeters**:

- 51 m standoff: `rel_pos = [+0.0018, 0, 1.125]` → y-axis `[0, +1, 0]`
- 52 m standoff: `rel_pos = [−0.0021, 0, 2.125]` → y-axis `[0, −1, 0]`

The x-component crosses zero between +1 m and +2 m of stretch, flipping the
whole tangent frame by 180°. The azimuth/elevation rotation
`R_azel = R_t_to_w * curr_R_t_to_w'` then carries a spurious 180° twist that
rotates the entire system. The existing degeneracy guard
(`wing_pos[1] ≈ 0.0 && wing_pos[2] ≈ 0.0`) never fired: `isapprox` against a
literal zero with default tolerances is *exact* equality.

### Defect 2 — `apply_heading!` blind to the flip
(`src/system_structure/transforms.jl`)

The heading step should have corrected any such twist (it solves for the
rotation that lands on `transform.heading`). But its no-ref-points branch
took `reference_body.R_b_to_w` — a property computed from `Q_b_to_w`, which
`apply_azimuth_elevation!` had **already rotated** — and then re-applied the
azel rotation to it (the code assumed the pre-rotation CAD orientation, per
its old docstring). The 180° twist thus entered the measured heading twice
and cancelled: the solve returned delta ≈ 0 and the flipped wing was never
corrected.

## Fix

1. **`apply_heading!`**: measure the current heading directly from the
   body's actual world orientation
   (`quaternion_to_rotation_matrix(reference_body.Q_b_to_w)`), without
   re-applying the azel rotation. The solved delta now always lands exactly
   on `transform.heading`, regardless of any twist picked up earlier in the
   transform chain. (Both callers are consistent with this: `reinit!`
   rotates `Q_b_to_w` in `apply_azimuth_elevation!` first, and
   `sim_reposition!` sets `transform.heading` explicitly before calling
   `reposition!`.)

2. **`calc_R_t_to_w`**: widen the degenerate-position guard from exact zero
   to near zero — `hypot(x, y) < 1e-2 * norm(pos)` (within 0.57° of zenith)
   pins the frame to the world-y convention, making the tangent frame
   deterministic for near-vertical positions.

Either fix alone resolves the crash; together they remove the spurious twist
at its source *and* make the heading step self-correcting against any future
upstream twist.

## Verification (HybridWings.jl ram_air_kite, elevation 76.5°)

| Tether | Before | After |
|---|---|---|
| 50 m | ok | heading 0.0°, AoA 13.49° (unchanged) |
| 51 m | ok | heading 0.0°, AoA 13.50° (unchanged) |
| 52 m | crash | heading 0.0°, AoA 13.50° |
| 100 m | crash | heading 0.0°, AoA 13.60° |
| 150 m | crash | heading 0.0°, AoA 13.64° |

At the 150 m figure-eight configuration, `find_steady_state!` settles to
heading −0.01° / AoA 9.4°, and 3 s of `next_step!` simulation with a
steady-torque hold runs stably.

Note for downstream users: configurations whose post-rotation heading was
not exactly the transform's target were previously left uncorrected
(delta ≈ 0 regardless); the heading step now actively enforces
`transform.heading`. For the A1-15 models the difference was ≈ 0° in
testing, but the first run after updating is worth a glance.

## Timeline / why it appeared now

The near-vertical CAD layout and both defects are old, but the flip boundary
sat outside the used parameter range until commit `090291b7` ("Fix aero
sections double-shifted", between v1.0.2 and v1.0.3) changed the ObjWing
COM/`T_cad_body` bookkeeping that the initial placement consumes — moving
the sign-crossing of the millimeter-scale horizontal offset to ~51–52 m of
standoff. 150 m runs worked on 2026-07-17 and crashed after the 1.0.3 bump
on 2026-07-18. (Note: a `main`-snapshot install can carry the fix while
still reporting v1.0.2 — the aero-fix commit landed just before the version
bump.)
